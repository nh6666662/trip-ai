# Harness Engineering（驾驭工程）完整架构详解

---

## 一、Harness Engineering 的定义

**Harness Engineering（驾驭工程）** 是一套围绕大模型构建的完整工程体系，目标是通过多层架构的协同，让大模型在有效的上下文约束下，持续规范地执行任务。

核心思想：**大模型只是引擎，Harness 是围绕引擎的一整套驾驭系统。**

## 三、各层详解

### 3.1 提示词工程（Prompt Engineering）

**定位**：控制大模型**说什么**、**怎么说**。

#### 核心要素

| 要素         | 作用                   | 示例                               |
| ------------ | ---------------------- | ---------------------------------- |
| **角色设定** | 定义模型身份           | "你是资深Python工程师"             |
| **背景**     | 任务背景、项目背景     | "我现在需要优化一段数据处理代码"   |
| **历史对话** | 多轮对话的连贯性       | "我前面提到xxxx"                   |
| **参考文档** | 官方文档、API文档      | "参照Python3官方文档"              |
| **限制条件** | 不要做什么、必须做什么 | "不改原有代码逻辑，只新增排序功能" |
| **输出格式** | 指定输出形式           | "直接返回完整可运行的函数代码"     |

#### 提示词工程的本质

提示词工程 = **有意识的调整你说话的内容和格式，让模型的输出稳定地朝着你预期的方向靠近。**

```
无引导的提示词 ──→ 模型 ──→ 实际输出（偏离预期）
     ↑                                    │
     └──────── 有意识的调整 ←─────────────┘
              （角色、背景、限制、格式）
```

---

### 3.2 上下文工程（Context Engineering）

**定位**：控制**什么时候**、**放什么内容**进上下文窗口。

#### 核心问题

大模型的上下文窗口是**有限的**。当对话轮数增多，上下文窗口会被填满，系统不得不丢弃或压缩部分信息，导致**上下文腐化**。

#### 上下文腐化的表现

- 模型"忘了"之前的约束
- 回答前后不一致
- 理解偏差，答非所问

#### 上下文工程的三个步骤

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│  召回   │ → │  压缩   │ → │  组装   │
│Retrieve │    │Compress │    │Assemble │
└─────────┘    └─────────┘    └─────────┘
```

##### 步骤 1：召回（Retrieve）

**问题**：获得什么信息？从哪里获取？

**信息来源**：

- 外部新闻、文档（RAG）
- 历史聊天记录（Memory）
- 当前代码文件
- 错误日志、报错信息

**关键**：不能随便拿一个信息就塞进去，需要**精准召回**与当前任务相关的信息。

##### 步骤 2：压缩（Compress）

**问题**：召回的信息太大，塞不进窗口怎么办？

**压缩方法**：

- 让大模型做**总结**（Summarization）
- 提取**关键信息**，丢弃冗余内容
- 分段压缩，保留核心语义

##### 步骤 3：组装（Assemble）

**问题**：信息顺序怎么排？

**组装原则**：

- **调整消息位置和顺序**
- 越靠后的信息，模型越重视（**近因效应**）
- 把**最重要的约束**放在靠后的位置

```
调整前：信息1 + 信息2 + 信息3
调整后：信息3 + 信息1 + 信息2  （关键信息放后面）
```

#### 上下文工程的本质

上下文工程 = **在合适的时候，把合适的内容，放进上下文窗口。**

---

### 3.3 记忆层（Memory Layer）

**定位**：解决**长期记忆**问题，确保核心信息不丢失。

#### 核心功能

- 保存项目目标、技术栈等**核心信息**
- 每次对话都**自动注入**关键约束
- 减少模型理解偏差

#### 实现方式

**方式 1：规则文件**

在项目根目录创建规则文件（如 `CLAUDE.md`、`.cursorrules`），作为系统提示词的一部分，每次对话自动加载。

**规则文件内容结构**：

```
[项目目标]
[技术栈]
[需求背景]
[代码风格]
[禁止事项]
[报错信息]
[聊天总结]
[提示词]
......
```

**方式 2：规则文件拆分**

当规则文件过长时，拆分成多个更短的文件：

```
src/
├── CLAUDE.md          # 主规则文件（索引作用）
├── bg.md              # 项目背景
├── stack.md           # 技术栈
├── yes.md             # 必须做的事
├── no.md              # 禁止做的事
└── rules/
    └── rule.md        # 具体规则
```

`CLAUDE.md` 作为索引：

```
背景看 bg.md
技术栈看 stack.md
需要做的事看 yes.md
禁止做的事看 no.md
做完跑 xx lint, xx 单测, CI, xx Skills
```

**方式 3：外部记忆数据库**

- 向量数据库（RAG）
- 对话总结（Summary）

#### 记忆层的作用

```
┌─────────────┐
│  规则文件   │──→ 系统提示词 ──→ 大模型
│  CLAUDE.md  │      ↑
└─────────────┘      │
                     │
              [项目目标、技术栈、代码风格、禁止事项]
              （每次对话自动注入，保证一直在）
```

---

### 3.4 执行层（Acting Layer）

**定位**：让大模型从"只能说话"变成"能真正干活"。

#### 核心能力

| 能力         | 说明                 | 工具      |
| ------------ | -------------------- | --------- |
| **执行命令** | 运行 Shell 命令      | Bash 沙箱 |
| **读写文件** | 读取、修改、创建文件 | 文件系统  |
| **外部工具** | 调用第三方 API、工具 | MCP       |

#### 执行层的组成

```
┌─────────────────────────────────────────┐
│              执行层（Acting）            │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │Bash沙箱 │  │文件系统 │  │  MCP   │ │
│  │执行命令 │  │读写文件 │  │外部工具 │ │
│  └─────────┘  └─────────┘  └─────────┘ │
└─────────────────────────────────────────┘
```

#### 执行层的工作流程

```
大模型思考（Reasoning）
        ↓
   决定执行什么
        ↓
   调用执行层
        ↓
   执行命令/读写文件/调用工具
        ↓
   返回执行结果
```

---

### 3.5 反馈层（Feedback Layer）

**定位**：收集执行结果，形成**试错循环**。

#### 核心功能

- 收集执行结果（成功/失败）
- 收集报错信息
- 将反馈加入下一轮上下文
- 形成**自我修正**能力

#### 反馈层的工作流程

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│ 记忆层  │ → │  大模型  │ → │ 执行层  │
│+上下文  │    │ 思考    │    │ 执行    │
└─────────┘    └─────────┘    └────┬────┘
       ↑                           │
       └──────── 反馈层 ←──────────┘
              （报错、结果、日志）
```

**关键**：执行失败时，报错信息被收集并注入下一轮上下文，模型根据报错修正策略，重新执行。

---

### 3.6 编排层（Orchestration Layer）

**定位**：控制整个 Agent 的运行流程，防止失控。

#### 核心功能

- 防止**无限死循环**
- 设定**明确的结束目标**
- 大任务**拆解**为多个小任务
- 按规划**分步执行**

#### 编排策略

**策略 1：任务拆解**

```
大任务
  ├── 小任务1：确认开发规范
  ├── 小任务2：编写接口代码
  ├── 小任务3：编写测试用例
  ├── 小任务4：执行代码校验
  └── 小任务5：验证功能闭环
```

**策略 2：分步执行**

按规划驱动 Agent 分步执行，形成**编排层**。

**策略 3：结束条件**

- 达到目标自动结束
- 超过最大循环次数强制结束
- 用户手动终止

#### 编排层的工具

**Spec-Kit**：开源工具，将需求拆成多个阶段

```
Constitution（生成约束） → Specification（明确需求）
         ↑                        ↓
Planning（定制计划） ←── Tasking（拆解任务）
         ↓
Implementation（开始实现）
```

每个阶段生成对应的约束文件，最后统一注入上下文。

## 四、ReAct 循环：思考 → 执行 → 反馈

### 4.1 ReAct 定义

**ReAct** = **Re**asoning（思考）+ **Act**ing（执行）

是 AI Agent 的核心工作模式。

### 4.2 ReAct 完整循环

```
┌─────────────────────────────────────────────────────────┐
│                      ReAct 循环                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   1. 上下文构建（记忆层 + 提示词工程 + 上下文工程）       │
│                     ↓                                   │
│   2. 大模型思考（Reasoning）                             │
│                     ↓                                   │
│   3. 执行层执行（Acting）                                │
│                     ↓                                   │
│   4. 反馈层收集结果（Feedback）                          │
│                     ↓                                   │
│   5. 反馈加入上下文，循环回到步骤 1                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.3 ReAct 循环的问题

每次循环，上下文都会**膨胀**。如果缺乏结束目标和清晰的执行标准，Agent 可能陷入**无限死循环**。

**解决方案**：引入**编排层**，控制循环次数和结束条件。

    大模型（LLM）
    + 提示词工程（Prompt Engineering）

    + 记忆层（Memory Layer）

## 六、Harness Engineering 解决的问题汇总

| 问题                   | 解决方案              | 对应层级   |
| ---------------------- | --------------------- | ---------- |
| 模型乱说话、输出不稳定 | 提示词工程            | 提示词工程 |
| 上下文太长、窗口溢出   | 召回 → 压缩 → 组装    | 上下文工程 |
| 模型失忆、前后不一致   | 规则文件 + 记忆层     | 记忆层     |
| 只能说话、不能干活     | Bash + 文件系统 + MCP | 执行层     |
| 无限循环、无法结束     | 任务拆解 + 结束条件   | 编排层     |
| 错误无法自修正         | 试错循环 + 反馈注入   | 反馈层     |

## 九、总结

Harness Engineering 不是单一技术，而是一套**完整的工程体系**：

1. **提示词工程** 控制模型说什么
2. **上下文工程** 控制什么时候放什么内容
3. **记忆层** 保证核心信息不丢失
4. **执行层** 让模型能真正干活
5. **反馈层** 形成试错和自我修正
6. **编排层** 控制整体流程不失控

**核心目标**：让大模型在有效的上下文约束下，持续规范地执行任务。

> 大模型只是引擎，Harness Engineering 是围绕引擎的一整套驾驭系统。
