# 旅智 TripAI — 项目总纲

---

## 一、项目概述

### 1.1 产品定位

AI 驱动的智能旅行规划平台，核心解决三大旅行痛点：规划难（手动规划耗时耗力）、变化多（计划赶不上变化）、信息滞后（实时人流/天气/交通不可知）。

### 1.2 核心价值主张

通过三个核心模块形成闭环：AI 辅助行程规划（降低规划门槛）→ 实时动态调整（应对突发变化）→ 用户 UGC 反馈（数据回流优化模型）。MVP 阶段目标是用最小成本验证"用户是否愿意使用 AI 动态行程调整"这一核心假设。

### 1.3 MVP 验证指标

| 指标           | 定义                         | 目标值   |
| -------------- | ---------------------------- | -------- |
| 行程创建率     | 新用户完成首个行程的比例     | > 40%    |
| 动态调整采纳率 | 用户接受系统调整建议的比例   | > 50%    |
| UGC 上报率     | 行程中用户主动上报数据的比例 | > 20%    |
| 行程分享率     | 创建行程后分享给旅伴的比例   | 辅助参考 |
| 7 日留存率     | 验证行程实用性与信息价值     | 辅助参考 |

### 1.4 三端产品矩阵

| 平台           | 定位                                    | 优先级   | 技术栈               |
| -------------- | --------------------------------------- | -------- | -------------------- |
| Web 官网       | 品牌展示、产品介绍、引导下载/注册       | 一期优先 | Next.js (SSR/SSG)    |
| Web 体验平台   | 完整产品体验（行程规划、AI 助手、社区） | 一期同步 | Next.js (App Router) |
| Flutter 移动端 | iOS + Android 原生体验 App              | 二期跟进 | Flutter + BLoC/Cubit |

**后端统一使用 Supabase**（PostgreSQL 数据库 + Auth 认证 + Storage 存储 + Realtime 实时推送），Web 端与移动端共享同一套后端服务。

---

## 二、功能规格

### 2.1 核心模块一：行程规划与管理

**用户操作流程**：输入目的地/日期/偏好 → AI 生成行程（提供"紧凑版"和"松弛版"两种节奏模板）→ 手动拖拽调整 → 导出/分享行程单。

**AI 生成能力**：调用高德 API（交通耗时、拥堵预测）、景点官方数据（开放时间）、第三方平台（餐厅评分）、气象 API，基于动态人流和天气数据优化时间分配。使用遗传算法生成多版本行程供选择。

**冲突检测**：修改行程时自动检测时间/交通冲突，给出调整建议（如"将午餐时间延后 30 分钟以避免堵车"）。

**MVP 简化**：AI 仅提供预置模板 + 简单偏好筛选，不做基于用户画像的深度定制；多人协作仅支持分享链接同步，不做实时协作编辑。

> 详细功能描述见 `功能.md` 第一章。

### 2.2 核心模块二：实时信息流与动态调整

**预警触发场景**：景点人流超阈值（排队超 30 分钟）、突发天气变化（暴雨预警自动替换户外行程至室内）、交通异常（地铁故障推荐替代方案含预估价格）。

**动态推荐**：基于当前位置 + 时间，推送附近高评分餐厅或当前时段人少的隐藏景点。

**用户决策**：用户可"立即采纳""查看备选""保留原计划"，系统记录拒绝原因用于后续优化。

**信息过滤**：仅推送高置信度且影响行程的关键信息（如拥堵导致延误超 30 分钟才提醒），避免信息过载。

> 详细功能描述见 `功能.md` 第二章。

### 2.3 核心模块三：用户 UGC 上传与验证

**上报入口**：进入景点时自动弹窗询问体验（1-5 星 + 可选文字）；地图页快捷按钮主动上报（人多/排队/设施故障 + 拍照）。

**验证机制**：地理围栏（定位需在景点 500m 内）+ 时间戳 + 多源交叉验证（多人同时上报拥堵）。可信度以三级标识展示："已验证" / "待审核" / "低可信"。

**数据回流**：验证后的 UGC 数据反馈至 AI 模型，优化后续行程推荐与实时预警，形成闭环。

## 四、架构设计

### 4.1 架构选型

一期 Web 端采用 **Next.js 14 App Router + Supabase** 全栈架构。选型理由如下：

Next.js App Router 提供 Server Components + Client Components 混合渲染能力，官网页面使用 SSG（静态生成）实现极致的 SEO 和加载速度，体验平台使用 CSR（客户端渲染）实现复杂交互。Supabase 作为开源 Firebase 替代品，提供 PostgreSQL 数据库、行级安全认证、对象存储和实时订阅，一套后端同时服务 Web 端和未来的 Flutter 移动端。

二期 Flutter 移动端采用 **Clean Architecture + BLoC/Cubit** 架构，通过 Supabase Dart SDK 直接对接后端，与 Web 端共享数据库和认证体系。

### 4.2 分层职责

```
Web 端（Next.js App Router）
├── Server Components — 服务端渲染，直接查询 Supabase，适合官网/SEO 页面
├── Client Components — 客户端交互，useState/useReducer 管理状态
├── API Routes (Route Handlers) — 服务端 API 端点，封装复杂业务逻辑
├── Server Actions — 表单提交、数据变更等服务端操作
└── Middleware — 认证拦截、路由守卫、国际化

后端（Supabase）
├── PostgreSQL — 关系型数据库，表结构设计 + RLS 行级安全策略
├── Auth — 用户认证（手机号、邮箱、OAuth 第三方登录）
├── Storage — 文件存储（UGC 图片、行程导出文件）
├── Realtime — 实时数据订阅（行程协作、预警推送）
├── Edge Functions — 服务端逻辑（AI 行程生成、UGC 验证）
└── RPC — 数据库函数（行程冲突检测、统计聚合）

Flutter 移动端（二期）
├── Presentation — Widget + BLoC/Cubit 状态管理
├── Domain — Use Cases + Entities（平台无关）
├── Data — Supabase Dart SDK 封装
└── Infrastructure — 设备能力（GPS、推送、本地缓存）
```

### 4.3 关键设计模式 — Web 端实现（Next.js + Supabase）

#### 4.3.1 观察者模式 — 实时预警推送

场景：用户行程中，交通异常、天气突变、景点人流拥堵时，实时推送预警信息。多个页面组件需要同时响应预警事件。

实现方式：利用 Supabase Realtime 订阅数据库变更，通过 React Hook 封装为可复用的 `useRealtimeAlerts`。

```typescript
// lib/hooks/use-realtime-alerts.ts
import { useEffect, useState, useCallback } from "react";
import { createBrowserClient } from "@/lib/supabase/client";
import type { RealtimeAlert, AlertPriority } from "@/types/database";

export function useRealtimeAlerts(tripId: string) {
  const supabase = createBrowserClient();
  const [alerts, setAlerts] = useState<RealtimeAlert[]>([]);

  useEffect(() => {
    const channel = supabase
      .channel(`alerts:${tripId}`)
      .on(
        "postgres_changes",
        {
          event: "INSERT",
          schema: "public",
          table: "realtime_alerts",
          filter: `trip_id=eq.${tripId}`,
        },
        (payload) => {
          const newAlert = payload.new as RealtimeAlert;
          // 仅推送高置信度预警，避免信息过载
          if (newAlert.confidence >= 0.7) {
            setAlerts((prev) => [newAlert, ...prev]);
          }
        },
      )
      .subscribe();

    return () => {
      supabase.removeChannel(channel);
    };
  }, [tripId, supabase]);

  const acceptSuggestion = useCallback(
    async (alertId: string) => {
      await supabase
        .from("realtime_alerts")
        .update({ status: "accepted" })
        .eq("id", alertId);
      setAlerts((prev) =>
        prev.map((a) =>
          a.id === alertId ? { ...a, status: "accepted" as const } : a,
        ),
      );
    },
    [supabase],
  );

  const dismissAlert = useCallback(
    async (alertId: string) => {
      await supabase
        .from("realtime_alerts")
        .update({ status: "dismissed" })
        .eq("id", alertId);
      setAlerts((prev) => prev.filter((a) => a.id !== alertId));
    },
    [supabase],
  );

  return { alerts, acceptSuggestion, dismissAlert };
}
```

#### 4.3.2 策略模式 — 多版本行程生成算法

场景：用户选择"紧凑版"或"松弛版"行程时，切换不同的时间分配策略。后续可能新增"亲子版""美食版"等策略。

#### 4.3.3 装饰器模式 — UGC 多维度可信度验证

场景：用户上报"景点排队 1 小时"，需结合地理围栏、时间戳、用户历史可信度等多维度验证真实性。

实现方式：Supabase Edge Function 内实现验证链，每个验证器只关注一个维度。

```typescript
// supabase/functions/validate-ugc/validators.ts

interface UGCValidator {
  validate(report: UGCReport): Promise<UGCValidationResult>;
}

interface UGCValidationResult {
  accepted: boolean;
  pending: boolean;
  rejected: boolean;
  confidence: number;
  note?: string;
}

// 基础验证器 — 检查必填字段
class BaseValidator implements UGCValidator {
  async validate(report: UGCReport): Promise<UGCValidationResult> {
    if (!report.content && report.photos.length === 0) {
      return {
        accepted: false,
        pending: false,
        rejected: true,
        confidence: 0,
        note: "内容不能为空",
      };
    }
    if (!report.spot_id) {
      return {
        accepted: false,
        pending: false,
        rejected: true,
        confidence: 0,
        note: "必须关联一个地点",
      };
    }
    return { accepted: true, pending: false, rejected: false, confidence: 0.3 };
  }
}

// 地理围栏验证器 — 装饰器
class GeoFenceValidator implements UGCValidator {
  private readonly MAX_DISTANCE = 500; // 米
  constructor(private inner: UGCValidator) {}

  async validate(report: UGCReport): Promise<UGCValidationResult> {
    const base = await this.inner.validate(report);
    if (base.rejected) return base;

    const distance = haversineDistance(
      report.user_lat,
      report.user_lng,
      report.spot_lat,
      report.spot_lng,
    );

    if (distance > this.MAX_DISTANCE) {
      return {
        ...base,
        accepted: false,
        pending: true,
        note: `距景点 ${Math.round(distance)}m，超出围栏`,
      };
    }

    return { ...base, confidence: Math.min(base.confidence + 0.3, 1) };
  }
}

// 时间一致性验证器 — 装饰器
class TimeConsistencyValidator implements UGCValidator {
  constructor(private inner: UGCValidator) {}

  async validate(report: UGCReport): Promise<UGCValidationResult> {
    const base = await this.inner.validate(report);
    if (base.rejected) return base;

    const ageHours =
      (Date.now() - new Date(report.created_at).getTime()) / 3600000;
    if (ageHours > 2) {
      return {
        ...base,
        confidence: base.confidence * 0.7,
        note: "上报超过 2 小时",
      };
    }

    if (new Date(report.created_at) < new Date(report.spot_opening_time)) {
      return {
        ...base,
        accepted: false,
        pending: true,
        confidence: base.confidence * 0.5,
        note: "早于开放时间",
      };
    }

    return base;
  }
}

// 用户信誉验证器 — 装饰器
class UserReputationValidator implements UGCValidator {
  constructor(
    private inner: UGCValidator,
    private supabase: SupabaseClient,
  ) {}

  async validate(report: UGCReport): Promise<UGCValidationResult> {
    const base = await this.inner.validate(report);
    if (base.rejected) return base;

    const { data: profile } = await this.supabase
      .from("user_profiles")
      .select("reputation_score")
      .eq("id", report.user_id)
      .single();

    const reputation = profile?.reputation_score ?? 0.5;
    if (reputation > 0.8) {
      return { ...base, confidence: Math.min(base.confidence + 0.2, 1) };
    }
    if (reputation < 0.3) {
      return {
        ...base,
        accepted: false,
        pending: true,
        confidence: base.confidence * 0.5,
        note: "用户信誉较低，建议人工复审",
      };
    }

    return base;
  }
}

// 组装验证链
function buildValidatorChain(supabase: SupabaseClient): UGCValidator {
  return new UserReputationValidator(
    new TimeConsistencyValidator(new GeoFenceValidator(new BaseValidator())),
    supabase,
  );
}
```

#### 4.3.4 仓储模式 — Supabase Client 封装

场景：行程数据需要同时支持 Supabase 查询和本地缓存（离线访问），且缓存策略可能随业务调整。

```typescript
// lib/supabase/trip-repository.ts
import { createBrowserClient } from "./client";
import type { Trip, TripNode } from "@/types/database";

export class TripRepository {
  private supabase = createBrowserClient();
  private cache = new Map<string, { trip: Trip; timestamp: number }>();
  private readonly CACHE_MAX_AGE = 6 * 60 * 60 * 1000; // 6 小时

  async getTrip(tripId: string): Promise<Trip | null> {
    // 1. 先查缓存
    const cached = this.cache.get(tripId);
    if (cached && Date.now() - cached.timestamp < this.CACHE_MAX_AGE) {
      return cached.trip;
    }

    // 2. 查询 Supabase
    const { data, error } = await this.supabase
      .from("trips")
      .select("*, trip_nodes(*)")
      .eq("id", tripId)
      .single();

    if (error) {
      // 3. 网络失败 → 兜底返回过期缓存
      return cached?.trip ?? null;
    }

    // 更新缓存
    this.cache.set(tripId, { trip: data, timestamp: Date.now() });
    return data;
  }

  async saveTrip(trip: Trip): Promise<Trip> {
    const { data, error } = await this.supabase
      .from("trips")
      .upsert(trip)
      .select()
      .single();

    if (error) throw error;
    this.cache.set(trip.id, { trip: data, timestamp: Date.now() });
    return data;
  }

  async getUserTrips(userId: string): Promise<Trip[]> {
    const { data, error } = await this.supabase
      .from("trips")
      .select("*, trip_nodes(*)")
      .eq("user_id", userId)
      .order("created_at", { ascending: false });

    if (error) throw error;
    return data ?? [];
  }
}
```

#### 4.3.6 Optimistic Updates — 行程编辑即时反馈

场景：用户拖拽调整行程节点顺序时，UI 需要即时响应，不等服务端确认。

```tsx
// app/(dashboard)/trips/[id]/timeline.tsx — Client Component
"use client";

import { useOptimistic, useTransition } from "react";
import { updateTripOrder } from "./actions";
import type { TripNode } from "@/types/database";

export function TripTimeline({
  tripId,
  initialNodes,
}: {
  tripId: string;
  initialNodes: TripNode[];
}) {
  const [optimisticNodes, setOptimistic] = useOptimistic(initialNodes);
  const [isPending, startTransition] = useTransition();

  const handleReorder = (oldIndex: number, newIndex: number) => {
    // 乐观更新：立即在 UI 上调整顺序
    startTransition(async () => {
      setOptimistic((prev) => {
        const updated = [...prev];
        const [item] = updated.splice(oldIndex, 1);
        updated.splice(newIndex, 0, item);
        return updated;
      });

      // 后台提交到服务端
      const result = await updateTripOrder(tripId, oldIndex, newIndex);
      if (result?.error) {
        // 服务端失败时，React 会自动回退到 initialNodes
        console.error("行程更新失败:", result.error);
      }
    });
  };

  return (
    <div className="space-y-2">
      {optimisticNodes.map((node, index) => (
        <TimelineNode
          key={node.id}
          node={node}
          isLast={index === optimisticNodes.length - 1}
          onDragEnd={(newIndex) => handleReorder(index, newIndex)}
          isPending={isPending}
        />
      ))}
    </div>
  );
}
```

### 4.4 Supabase 数据库设计

#### 核心表结构

```sql
-- 用户表（扩展 Supabase Auth 的 auth.users）
CREATE TABLE user_profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT,
  avatar_url TEXT,
  travel_preferences TEXT[],          -- 偏好标签：'亲子游', '自然风光', '美食'
  reputation_score FLOAT DEFAULT 0.5, -- UGC 信誉分 0-1
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 行程表
CREATE TABLE trips (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
  destination TEXT NOT NULL,
  departure TEXT,
  start_date DATE NOT NULL,
  end_date DATE NOT NULL,
  pace TEXT DEFAULT 'relaxed',        -- 'tight' | 'relaxed'
  traveler_count INT DEFAULT 1,
  preferences TEXT[],
  status TEXT DEFAULT 'draft',        -- 'draft' | 'confirmed' | 'completed'
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 行程节点表
CREATE TABLE trip_nodes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  spot_id UUID REFERENCES spots(id),
  name TEXT NOT NULL,
  node_type TEXT DEFAULT 'spot',      -- 'spot' | 'meal' | 'rest' | 'transit' | 'custom'
  start_time TIMESTAMPTZ NOT NULL,
  duration_minutes INT NOT NULL,
  transit_minutes INT DEFAULT 0,
  sort_order INT NOT NULL,
  metadata JSONB DEFAULT '{}'
);

-- 景点表
CREATE TABLE spots (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  description TEXT,
  image_url TEXT,
  latitude DOUBLE PRECISION NOT NULL,
  longitude DOUBLE PRECISION NOT NULL,
  min_visit_minutes INT DEFAULT 60,
  recommended_minutes INT DEFAULT 120,
  rating FLOAT DEFAULT 0,
  tags TEXT[],
  opening_time TIME,
  closing_time TIME
);

-- UGC 上报表
CREATE TABLE ugc_reports (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
  spot_id UUID NOT NULL REFERENCES spots(id),
  trip_id UUID REFERENCES trips(id),
  content TEXT,
  photos TEXT[],                      -- Supabase Storage 路径数组
  rating INT CHECK (rating BETWEEN 0 AND 5),
  user_lat DOUBLE PRECISION,
  user_lng DOUBLE PRECISION,
  status TEXT DEFAULT 'pending',      -- 'pending' | 'verified' | 'rejected'
  confidence FLOAT,
  upvotes INT DEFAULT 0,
  comment_count INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 实时预警表
CREATE TABLE realtime_alerts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  trip_id UUID NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  alert_type TEXT NOT NULL,           -- 'traffic' | 'weather' | 'crowd' | 'facility'
  priority TEXT DEFAULT 'medium',     -- 'high' | 'medium' | 'low'
  title TEXT NOT NULL,
  description TEXT,
  confidence FLOAT NOT NULL DEFAULT 0.5,
  suggestion TEXT,
  status TEXT DEFAULT 'pending',      -- 'pending' | 'accepted' | 'dismissed'
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 索引
CREATE INDEX idx_trips_user ON trips(user_id);
CREATE INDEX idx_trip_nodes_trip ON trip_nodes(trip_id);
CREATE INDEX idx_ugc_spot ON ugc_reports(spot_id);
CREATE INDEX idx_alerts_trip ON realtime_alerts(trip_id);

-- RLS 行级安全策略
ALTER TABLE trips ENABLE ROW LEVEL SECURITY;
ALTER TABLE trip_nodes ENABLE ROW LEVEL SECURITY;
ALTER TABLE ugc_reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE realtime_alerts ENABLE ROW LEVEL SECURITY;

-- 用户只能访问自己的行程
CREATE POLICY "Users can view own trips" ON trips FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "Users can modify own trips" ON trips FOR ALL USING (auth.uid() = user_id);

-- UGC 公开可读，仅作者可改
CREATE POLICY "UGC readable by all" ON ugc_reports FOR SELECT USING (true);
CREATE POLICY "UGC modifiable by author" ON ugc_reports FOR ALL USING (auth.uid() = user_id);
```

### 4.5 状态管理定义

**Web 端（Next.js）**：

| 模块       | 方案                            | 说明                                       |
| ---------- | ------------------------------- | ------------------------------------------ |
| 全局状态   | Zustand 或 Context + useReducer | 轻量级，适合行程数据、用户会话             |
| 服务端状态 | React Query (TanStack Query)    | 缓存 Supabase 查询结果，自动重取、乐观更新 |
| 表单状态   | React Hook Form + Zod           | 行程创建/编辑表单验证                      |
| 实时数据   | Supabase Realtime + useEffect   | 预警推送、行程协作同步                     |

**Flutter 端（二期）**：

| 模块       | 类型  | 事件/方法                                       | 状态                                                        |
| ---------- | ----- | ----------------------------------------------- | ----------------------------------------------------------- |
| 行程规划   | BLoC  | `TripGenerateRequested`, `TripNodeReordered`    | `Initial` / `Loading` / `Loaded(trip, conflicts)` / `Error` |
| 实时预警   | Cubit | `startWatching(tripId)`, `acceptSuggestion(id)` | `RealtimeAlertState(alerts: [])`                            |
| AI 对话    | BLoC  | `MessageSent`, `QuickActionTriggered`           | `Initial` / `Streaming` / `Loaded(messages)` / `Error`      |
| UGC 上传   | Cubit | `init(spotId)`, `submit()`                      | `Idle` / `Submitting` / `Success(result)` / `Error`         |
| 社区信息流 | Cubit | `loadFeed()`, `loadMore()`, `switchTab(tab)`    | `Loading` / `Loaded(items, hasMore)` / `Error`              |

### 4.6 业务模块数据流

**行程规划（Web 端）**：

```
用户输入表单 → Server Action / API Route
  → Edge Function: generate-trip（策略模式选择算法）
  → Supabase: 查询 spots + weather + traffic
  → 生成 Trip + TripNodes → INSERT 到 trips/trip_nodes 表
  → 返回 Trip 数据 → Client Component 渲染 TripTimeline
```

**实时预警（Web 端）**：

```
Supabase Edge Function / Cron: 定时检测交通+天气+UGC 数据
  → 工厂模式统一格式 → INSERT 到 realtime_alerts 表
  → Supabase Realtime 触发 WebSocket 推送
  → useRealtimeAlerts Hook 接收 → 按 priority 渲染预警卡片
```

**UGC 验证（Web 端）**：

```
用户提交表单 → Server Action
  → Edge Function: validate-ugc（装饰器验证链）
  → Base → GeoFence → TimeConsistency → UserReputation
  → confidence >= 0.7: UPDATE status='verified'
  → 0.3~0.7: UPDATE status='pending'
  → < 0.3: 不写入，提示用户
```

### 4.7 Supabase 客户端初始化

Web 端使用 `@supabase/supabase-js` 创建全局 Supabase Client 实例，通过环境变量配置连接信息。不同运行环境使用不同的 Client 工厂：

```typescript

  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
  );
}

// lib/supabase/server.ts — Server Client（Server Components 使用）
import { createServerComponentClient as createClient } from "@supabase/auth-helpers-nextjs";
import { cookies } from "next/headers";
export function createServerComponentClient() {
  return createClient({ cookies });
}

// lib/supabase/middleware.ts — Middleware Client（路由守卫使用）
import { createMiddlewareClient as createClient } from "@supabase/auth-helpers-nextjs";
export function createMiddlewareClient(req, res) {
  return createClient({ req, res });
}
```

Flutter 端（二期）使用 `supabase_flutter` 包，通过 `Supabase.initialize()` 在 `main()` 中初始化全局单例。

### 4.8 设计陷阱与成本控制

**严格禁止的行为**：

Client Component 中直接查询数据库——所有涉及 RLS 策略的查询必须通过 Server Component 或 API Route 完成，确保行级安全策略正确执行。

滥用 `'use client'` 指令——默认所有组件都是 Server Component，仅在需要用户交互（事件监听、状态管理、浏览器 API）时才标注 `'use client'`。每多一个 Client Component 都会增加客户端 JS 体积。

Edge Function 中做重计算——Edge Function 有执行时间限制（免费层 2 秒），AI 行程生成等耗时操作应使用异步队列或在 API Route 中处理。

**成本控制关键点**：

MVP 阶段 Supabase 使用免费层（500MB 数据库、1GB Storage、2GB 带宽、5 万月活用户）。Edge Function 调用量限制为每月 50 万次。第三方 API 调用（高德、和风天气）通过 Edge Function 代理，便于统一限流和缓存。

**架构演进路线**：

```
MVP（当前）              V1.0                       V2.0
─────────────          ─────                      ─────

免费层                  + Supabase Pro              + 离线优先策略
手动迁移 SQL            + supabase/db 版本化        + 多语言国际化
核心组件单测            + Sentry 错误监控           + AI 个性化推荐
```

## 八、开发规范

### 8.1 代码规范

**TypeScript / Next.js**：ESLint + Prettier 统一代码风格。文件命名使用 `kebab-case`（如 `trip-timeline.tsx`），组件名使用 `PascalCase`（如 `TripTimeline`）。Server Components 默认不加 `'use client'` 指令，仅在需要客户端交互时显式标注。所有 Supabase 查询使用自动生成的 TypeScript 类型。Zod schema 与 TypeScript 类型保持单一数据源。

**Flutter（二期）**：遵循 `flutter_lints` 官方推荐规则。Widget 文件使用 `snake_case`，类名使用 `PascalCase`。BLoC 事件/状态使用 sealed class。

### 8.2 Git 规范

分支策略：`main`（生产）→ `develop`（开发）→ `feature/*`（功能分支）。

提交信息格式：`<type>(<scope>): <description>`。type 取值：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`style`（样式）、`test`（测试）、`chore`（工程化）。scope 取值：`trip`、`realtime`、`ugc`、`ai`、`community`、`core`。

### 8.3 设计交付规范

Web 端设计稿以 1440px 宽度为基准出图，附带 375px 移动端适配标注。移动端设计稿以 390×844pt（iPhone 14/15）为基准。

所有颜色引用 Token 名称，所有字号引用 Token 名称，间距使用 4px 倍数。组件命名：Web 端 BEM（`trip-timeline__node--active`），Flutter 端 PascalCase（`TripTimelineNode`），两端共享语义。

## 九、风险与约束

### 9.1 技术风险

| 风险                | 影响                                             | 缓解措施                                        |
| ------------------- | ------------------------------------------------ | ----------------------------------------------- |
| 多源数据一致性      | 景点临时关闭/时间变更未同步                      | 缓存设置 6 小时过期 + 用户 UGC 交叉验证         |
| 行程冲突检测复杂度  | 调整后全局重算耗时                               | MVP 仅检测相邻节点冲突，V1.0 实现全局优化       |
| Supabase 免费层限额 | 用户量超预期后服务降级（数据库 500MB、带宽 2GB） | 监控用量仪表盘，随时可升级 Pro 计划（$25/月）   |
| AI 模型响应延迟     | 行程生成等待过长                                 | 骨架屏 + 打字机动画掩盖延迟，超时降级为模板行程 |

### 9.2 合规约束

所有功能设计需符合《个人信息保护法》与 GDPR（若面向海外市场）。用户 UGC 使用规则需明确告知并获取授权。位置数据采集需说明用途并提供关闭选项。
